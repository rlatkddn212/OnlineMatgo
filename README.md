### 네트워크 구성

![1560471835672](https://github.com/rlatkddn212/OnlineMatgo/blob/master/assets/1560471835672.png)

(mongo.db는 사용 못함)

Restful API 구조로 설계하여 유니티 클라이언트에서 HTTP 메시지를 보냅니다.

MongoDB에 영구적으로 저장될 데이터를 저장하고 Redis에 임시적으로 저장될 데이터를 저장하게 됩니다.



**restful** **구조**

Json 형식으로 데이터 통신이 가능함 (Node.js단에서 파싱에 오버헤드가 줄어듬)

상태 저장이 없음. 분산 서버 구성이 유리해짐

HTTP 기본 메소드 ( get, push, put, delete)로 접근. 순수 웹 기술들을 사용할 수 있음

soap 보다 간편(구조 구현 부분)

NoSQL, Node.js 콤비로 구현하기 편함



**long polling** **구조**

http는 클라이언트에서 요청하면 응답을 해주는 서버라서 서버가 실시간으로 중계할 수 없습니다.

클라이언트가 서버에 계속 요청을 보내는 폴링으로 서버를 구성할 수 있지만 서버에 부하가 많이 발생하게 되고 이런 HTTP의 단방향 통신을 해결한 방법으로 **long polling(comet)**을 활용 할 수 있습니다.



**몇 가지 문제점**

 

요청응답(HTTP)으로 모든 통신처리를 하게 됩니다. 두 유저가 방에 접속하여 통신을 하게 되면 실시간으로 대전하게 되는데 HTTP 통신으로 실시간 통신을 구현하려면 Polling 혹은 long Polling을 해야 합니다. (웹소켓도 있음)

Long polling으로 구현 할 때 문제점은 서버에서 상태를 저장 하지 않는 다는 점입니다.

만약 유저가 방에 접속하게 되면 서버에서 redis에 유저를 저장시키고 방 이름(혹은 방 번호)로 구독자를 등록시킵니다. (redis의 발행자/구독자(pub/sub) 라는 기술을 통해 이벤트를 주고 받을 수 있음) Socket을 활용하면 Connect를 유지 하기 때문에 상태를 저장할 수 있고 쉽게 발행자 구독자를 활용할 수 있습니다.

하지만 Long Polling은 서버에서 응답 메시지를 저장 시켜 두었을 때는 문제없지만 그렇지 않은 상태에서 메시지를 받는다면 메시지를 못 받을 수 있습니다. 이를 대비하여 메시지를 저장하는 큐를 구현해야 할 것 같습니다. 다시 접속한다고 해서 같은 서버에 들어 올리도 없고. 그래서 생각한게 redis 안에 메시지 큐를 만드는 방법을 택했습니다.

![1560472241300](https://github.com/rlatkddn212/OnlineMatgo/blob/master/assets/1560472241300.png)
